## 1. Co potwierdziłem w źródłach

BDOS publicznie opisuje się jako **engine między Claude Code i Google Ads API**, a nie paczka promptów/markdownów. Ich LinkedIn mówi o account diagnostics, search-term analysis, campaign creation, preview before every change, PDF reports, Google Sheets export, workflowach typu `/bdos-daily-check`, `/bdos-monthly-review`, `/bdos-keyword-research`, lokalnym działaniu i wbudowanej eksperckiej wiedzy Ads. To potwierdza nasz kierunek: **nie prompt pack, tylko engine + API + operator runtime**. ([LinkedIn][2])

Google Ads API pozwala robić realne mutacje i nawet tworzyć kampanię, ad groupy, reklamy i keywordy w jednym request chainie przez temporary resource names. To oznacza, że pełny write layer jest technicznie realny od startu. ([Google for Developers][3]) Google ma też Recommendations API i optimization score na poziomie konta/kampanii, więc Ads Doctor powinien pobierać rekomendacje Google, ale filtrować je przez nasz expert engine, a nie ślepo aplikować. ([Google for Developers][4])

GSC API daje dokładnie dane, których potrzebujemy do SEO/content intelligence: query, page, clicks, impressions, CTR i position. Oficjalne przykłady pokazują query/page analytics po zakresach dat, więc GSC może być rdzeniem CTR opportunities, content decay, cannibalization i query-page map. ([Google for Developers][5])

Ahrefs API v3 daje Site Explorer, Keywords Explorer, Site Audit, Rank Tracker, SERP Overview, Batch Analysis, Brand Radar i Social Media Management endpoints. Czyli Ahrefs nie jest tylko “keyword volume”, tylko realna warstwa do content gap, competitor gap, backlink gap, keyword clustering i site audit. ([Ahrefs for Developers][6])

Localo ma oficjalną dokumentację API & MCP Integration z 25 maja 2026. Opisują MCP jako kontrolowane połączenie AI z Localo, dostęp do rankingów keywordów, widoczności GBP względem konkurencji, trendów SEO i raportów. Czyli Localo wrzucamy jako pełny connector, a nie poboczny dodatek. ([Localo][7])

WordPress REST API oficjalnie pozwala czytać, modyfikować i tworzyć treści jako JSON przez REST endpoints. To potwierdza, że ekologus.pl i sklep.ekologus.pl mogą być pełnymi read/write źródłami content inventory i publikacji. ([WordPress Developer Resources][8])

LinkedIn Posts API pozwala publikować/comment/like jako organizacja przy odpowiednich rolach i permission `w_organization_social`, a Facebook Pages API dokumentuje tworzenie, publikowanie i aktualizowanie postów na Page. To znaczy, że Social Publisher jest realny, ale zależy od permissionów i app access. ([Microsoft Learn][9])

Codex ma dokładnie te primitives, których potrzebujemy: Skills, Hooks, Subagents, `codex exec`, `AGENTS.md` i `/goal`. Skills są folderami z `SKILL.md`, opcjonalnymi `scripts/`, `references/` i `assets/`, więc możemy robić cienkie skillsy, które wołają WILQ API zamiast trzymać całą logikę w promptach. ([OpenAI Developers][10]) Hooks pozwalają odpalać deterministyczne skrypty w lifecycle Codexa, np. logging, blokady sekretów, walidacje na końcu tury i directory-specific prompting. ([OpenAI Developers][11]) Subagents są oficjalnie dostępne i służą do równoległych workflowów dla złożonych tasków, ale Codex odpala je tylko, gdy wyraźnie o to poprosisz; zużywają więcej tokenów, więc używamy ich celowo, ale agresywnie. ([OpenAI Developers][12])

Załączony brief mówił wprost: WILQ jako marketer, Codex CLI/Desktop jako daily runtime, pełne API, lightweight, dynamiczny dashboard, skillsy ze `references/scripts`, realne metryki i użyteczne testy przez `codex exec`, bez statycznych HTML/artifactów i bez testów-dowodów. To zostaje fundamentem.

## 2. Finalny stack technologiczny

| Warstwa         | Decyzja                                                    | Dlaczego                                                                                                                                                                                                             |
| --------------- | ---------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Backend API     | **FastAPI + Pydantic**                                     | Szybkie API, OpenAPI z automatu, walidacja kontraktów, dobry fit pod Pythonowe SDK Google/Ahrefs/ETL. FastAPI oficjalnie pozycjonuje się jako szybki framework na Starlette + Pydantic. ([fastapi.tiangolo.com][13]) |
| Data validation | **Pydantic v2**                                            | Każdy action object, connector response, opportunity i Codex output musi mieć schema. Pydantic jest szeroko używaną biblioteką walidacji danych w Pythonie. ([pydantic.dev][14])                                     |
| Operational DB  | **SQLite**                                                 | Lokalny produkt, szybki start, mały operational state: actions, approvals, connector status, audit, codex runs.                                                                                                      |
| Analytics DB    | **DuckDB**                                                 | Metryki Ads/GSC/Ahrefs/Localo trzeba joinować, agregować i porównywać po okresach. DuckDB jest in-process SQL OLAP DB z Python clientem i świetnie pasuje do lokalnej analityki. ([DuckDB][15])                      |
| Background jobs | **Dramatiq/RQ + APScheduler, później Redis jeśli trzeba**  | Refresh connectorów, harmonogramy, długie pull-e Ahrefs/GSC/Ads. Lokalnie na start może być SQLite-backed/job table, ale projektujemy interfejs pod worker.                                                          |
| Frontend        | **Vite + React + TypeScript**                              | Vite to szybkie frontend tooling z dev serverem/HMR; React daje dynamiczny command center, nie statyczny raport. ([vitejs][16])                                                                                      |
| Server state UI | **TanStack Query**                                         | Dashboard będzie żył z API, freshness, pollingiem, mutations i invalidation. TanStack Query jest dokładnie do fetch/cache/update/observe server state w TS/JS. ([tanstack.com][17])                                  |
| Routing UI      | **TanStack Router**                                        | Nested command center, detail panels, query params, deep links do opportunity/action.                                                                                                                                |
| UI components   | **shadcn-style + Radix + Tailwind**                        | Szybki enterprise UI bez robienia design systemu od zera.                                                                                                                                                            |
| Charts/tables   | **Recharts + TanStack Table**                              | Metryki, trends, sortable dense tables, drilldowny.                                                                                                                                                                  |
| CLI             | **Typer**                                                  | `wilq dev`, `wilq connectors status`, `wilq ads doctor`, `wilq codex smoke`.                                                                                                                                         |
| Codex runtime   | **Codex Desktop/CLI + Skills + Hooks + Subagents + /goal** | To jest daily runtime WILQ-a i executor repo.                                                                                                                                                                        |
| MCP             | **Adapter, nie core**                                      | Localo MCP i ewentualne MCP są wejściem do systemu, ale core to nasze WILQ API, żeby dashboard i Codex miały jedno źródło prawdy.                                                                                    |
| Secrets         | **.env + local keyring później**                           | Na teraz `.env`, ale z redaction i secret scan hooks.                                                                                                                                                                |
| Tests           | **pytest + vitest + codex exec schema smoke**              | Użyteczne testy: kontrakty, reguły, connector status, output schema.                                                                                                                                                 |

W skrócie:

```txt
Python/FastAPI = mózg i connector engine
DuckDB = analityka marketingowa
SQLite = pamięć operacyjna
Vite/React/TS = dynamiczny cockpit
Codex Skills/Hooks/Subagents = runtime wykonawczy
```

## 3. Gdzie zmierzamy

Docelowo WILQ rano odpala Codex Desktop albo dashboard i widzi nie “metryki”, tylko **operacyjny stan marketingu Ekologus**:

```txt
1. Co przepala budżet w Adsach.
2. Co daje największą szansę na ruch z GSC.
3. Jakie frazy z Ads powinny zostać SEO landingiem.
4. Jakie treści już istnieją i nie wolno ich dublować.
5. Co trzeba przepisać pod CTR, intent, limity znaków i ton.
```

## 5. Action model — kręgosłup produktu

Nie “raport”, tylko obiekty akcji:

```json
{
  "id": "act_2026_06_17_001",
  "title": "Dodaj negative keywords dla zapytań bez konwersji",
  "domain": "google_ads",
  "connector": "google_ads",
  "mode": "apply",
  "risk": "medium",
  "evidence_ids": ["ads_search_terms_...", "campaign_..."],
  "metrics": [
    { "name": "cost", "value": 842.12, "period": "last_28_days" },
    { "name": "conversions", "value": 0, "period": "last_28_days" }
  ],
  "human_diagnosis": "Kampania płaci za zapytania, które nie pasują do intencji klienta Ekologus.",
  "payload": {
    "campaign_id": "...",
    "negative_keywords": ["..."]
  },
  "status": "ready"
}
```

Każda karta w dashboardzie i każdy skill Codexa musi działać na tym kontrakcie.

Twarda reguła:

```txt
No evidence_id → no recommendation.
No source connector → no recommendation.
No validated payload → no apply.
No audit event → no write.
```

## 7. Jak tworzyć idealne `/goal`

Z dokumentacji OpenAI wyciągam taki format:

```txt
1. Objective — co ma być dowiezione.
2. Non-goals — czego nie ruszać.
3. Constraints — stack, granice, styl, bezpieczeństwo.
4. Subagent strategy — kogo odpalić i po co.
5. Deliverables — konkretne pliki/moduły/endpointy/widoki.
6. Validation — komendy i testy.
7. Stop condition — kiedy Codex ma przestać.
```

To jest dokładnie zgodne z oficjalnym opisem `/goal`: osiągnąć konkretny cel, nie zmieniać rzeczy spoza zakresu, walidować progres i wiedzieć, kiedy stop. ([OpenAI Developers][1])

## 8. Gotowy `/goal` do odpalenia

To jest pierwszy goal. Nie “MVP”. To jest **pierwszy final-product architecture increment**: rdzeń API, modele, connectory registry, action engine, dashboard shell, Codex runtime. Bez tego każdy kolejny moduł będzie slopem.

```txt
/goal Build Goal 0012A: WILQ Marketing Operating System Core for Ekologus.

Objective:
Build the final-product core architecture for WILQ Marketing Operating System, a local API-first marketing command center for Ekologus. This system must connect a dynamic dashboard, Codex Desktop/CLI runtime, full API connector architecture, expert marketing rules, and real action objects. Do not frame this as MVP, demo, artifact, or static report. Build production-shaped foundations from the first commit.

Product context:
- WILQ is our marketer.
- Ekologus is the only client for this product.
- Codex Desktop/CLI is WILQ’s daily runtime.
- Dashboard and Codex skills must use the same local WILQ API.
- We need full API-first architecture for Google Ads, Google Search Console, Ahrefs, Localo, WordPress, LinkedIn, and Facebook.
- The product must turn raw metrics into marketer-readable diagnosis and executable actions.
- The dashboard must be dynamic, dense, useful, and operational.
- No recommendation may exist without a source connector, evidence id, raw metric/source fact, plain-language diagnosis, recommended action, risk, and action mode.
- No Codex skill may invent metrics. It must call WILQ API.

Confirmed external constraints:
- Google Ads API supports real mutate operations and recommendations, so design real read/write action support.
- GSC provides clicks, impressions, CTR, and position by query/page/date, so design SEO/content opportunity rules around those metrics.

- Backend: Python, FastAPI, Pydantic v2, SQLModel or SQLAlchemy, httpx, Typer.
- Operational state: SQLite.
- Analytical metric store: DuckDB.
```
