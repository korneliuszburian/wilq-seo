# 3. Skillsy są wartościowe, ale nie są wykorzystywane jak trzeba

Tak: skillsy mogą dawać realną wartość. Ale repo używa ich bardziej jako **operator/dev workflow + eval harness**, a nie jako **compiled product capability**.

Dobre:

* skills wymuszają polski output,
* skills wymuszają source/evidence/source connector guardrails,
* skills mogą trzymać sposób myślenia eksperta,
* skill eval ledger jest dobrym anti-slop narzędziem.

Ale problem jest taki:

> Skill nie może być głównym mózgiem produktu, jeśli produkt ma być WILQ API-first.

Czyli każde wartościowe zachowanie ze skilla musi zostać przeniesione do jednego z trzech miejsc:

1. **Domain rule / expert rule**
   np. „nie rekomenduj negatywnego keyworda bez 90-day cross-check”.

2. **Workflow contract**
   np. `/wilq-daily-check` zwraca `anomalies`, `blocked_checks`, `safe_next_actions`, `evidence_ids`.

3. **Safety/eval fixture**
   np. test: „jeśli search term miał konwersję w ostatnich 90 dniach, nie może być auto-kandydatem do wykluczenia”.

Skillsy mają zostać **źródłem ekstrakcji eksperckich wzorców**, nie miejscem, gdzie produkt „wie”.

# 4. Jak powinna wyglądać prawdziwa warstwa knowledge

Trzeba rozdzielić cztery typy wiedzy.

## A. Client Truth Knowledge

To jest: kim jest Ekologus, jakie ma usługi, czego nie obiecuje, jakie ma CTA, jakie claimy są legalnie ryzykowne.

To macie zaczęte, ale płytkie. Service Profile mówi: 10 knowledge cards, 7 service cards, **approved-current 0**, **production-depth 0**, private proposals 5, review actions 13. ([GitHub][5])

To jest potrzebne dla contentu, ale nie wystarczy do BDOS-class.

## B. Expert Operating Knowledge

To jest najważniejsza brakująca warstwa.

Przykłady:

* jak diagnozować spadek ROAS,
* kiedy CPA alert jest fałszywy,
* kiedy PMax przepala budżet,
* kiedy feed title jest słaby,
* kiedy Merchant issue jest krytyczny,
* jak interpretować GA4 traffic quality,
* jak porównać seasonal drop YoY,
* jak robić negative keyword review,
* kiedy nie ufać API/UI,
* kiedy measurement window jest za krótkie.

To musi być zakodowane jako **rules/playbooks/workflows**, a nie jako markdown.

## C. Platform/API Trap Knowledge

BDOS ma tutaj dużą przewagę.

To są rzeczy typu:

* GAQL structure, SELECT/FROM/WHERE/LIMIT, field compatibility, query cost. Google Ads API dokumentuje query clauses i rekomenduje selektywne pobieranie pól, LIMIT w dev/test, filtrowanie w WHERE oraz sprawdzanie kompatybilności pól przez GoogleAdsFieldService. ([Google for Developers][6])
* validate_only / partial failure / warnings / bulk mutate. Google Ads API ma `validate_only`, partial failures i bulk mutate patterns, ale nie każdy serwis wspiera partial failure, więc system musi to wiedzieć przed mutacją. ([Google for Developers][7])
* GA4 sampling/quotas/dimensions/metrics. GA4 Data API ma schema dimensions/metrics, quotas i reporting-data expectations; tego nie wolno traktować jak magicznych liczb bez provenance. ([Google for Developers][8])
* Merchant item issues, statuses, severity, destinations, title rules. Merchant/Content API pozwala czytać product statuses i item-level issues, a Google Merchant docs definiują title/product data best practices. ([Google for Developers][4])

WILQ powinien mieć te rzeczy jako **API-owned trap rules**.

## D. Account / Workspace Memory

To jest „folder per konto” z BDOS:

* business brief,
* exclusions,
* previous decisions,
* mutation history,
* reports,
* source packs,
* known false positives,
* client preferences,
* last review,
* open blockers.

WILQ ma docs/handoffs/beads, ale nie ma jeszcze normalnego `WorkspaceMemory` albo `AccountDossier`.

# 12. Realny BDOS-class direction dla WILQ

WILQ nie powinien kopiować BDOS 1:1, bo BDOS jest Google Ads/PPC-heavy.

Ale WILQ powinien skopiować **klasę systemu**:

## BDOS pattern

```text
command -> evidence fetch -> expert rules -> safe recommendation -> optional mutation preview -> audit -> report
```

## WILQ pattern


## Python debt

Najgorsze miejsca: `wilq/actions/service.py`, `wilq/schemas.py`, `wilq/briefing/content_diagnostics.py`. Nie chodzi o estetykę. Chodzi o to, że nowe product capabilities będą kusząco łatwe do dopisania w centralnych plikach, przez co każdy następny Goal zwiększy blast radius. Complexity script dobrze to rozpoznaje, ale sam guard nie usuwa długu. ([GitHub][18])

Dashboard jest dużo lepszy dzięki `surfaceRegistry.ts` i shared schemas, ale `ContentWorkflowSurface.tsx` ma 2223 linie. To jest za duża powierzchnia na ekran, który ma tylko renderować API-owned view-models. `PLAN.md` mówi jasno: API/schema/view-model/domain ma posiadać visible text i decyzje; route components mają renderować, nie remapować zachowań. ([GitHub][21])

## Test theater risk

Repo ma imponującą liczbę testów i skill evals, ale ryzyko teatru jest realne, bo dużo proofów dotyczy **shape, blockers, dashboards, packets, smoke**. Goal 005 completion audit mówi wprost: no fresh full verify for completion, real Wilku UAT incomplete, no owner-defer artifact. To jest właściwa samoocena, ale tylko jeśli owner jej nie zignoruje. ([GitHub][3])

## API router/domain boundaries

API wygląda zdrowiej niż typowy slop dashboard. Routery są rozdzielone: actions, codex, connectors, content_workflow, demand_gen, diagnostics, evidence, expert, jobs, knowledge, metrics, opportunities, social, system, workflows. To jest dobry kierunek. Słaby punkt: `content_workflow.py` może rosnąć jako endpoint aggregator, a domena nadal ciąży ku dużym centralnym plikom. ([GitHub][23])

Skill ledger ma sensowne hard gates: Polish operator language, source connectors/evidence, no invented metrics, blocked unsupported claims. ## Docs drift / docs bloat

`docs/PROGRESS.md` ma ponad 4000 linii i setki KB, a `docs/evals/skill-eval-ledger.md` ponad 10 tys. linii. To już nie jest lekki recovery ledger, tylko osobny system pamięci. Beads i docs są bogate, ale grożą sytuacją: „prawda jest gdzieś w trzech plikach i dwóch audytach”. ([GitHub][16])

## Beads/docs consistency

Dobra wiadomość: Beads trzyma otwarte `wilq-seo-jst` dla real Wilku content UAT/defer; completion audit mówi, żeby Goal 005 nie zamykać. Zła wiadomość: dużo zadań jest zamykanych wokół polish/review cards/readiness wording, co może dawać poczucie ruchu bez zamknięcia core blockerów. Ostatnie commity z 2026-07-04 to m.in. „raise Localo/GSC operator to Wilku-ready”, review cards i task closure — wygląda jak polishing readiness surfaces, nie jak dowiezienie write/apply albo production-depth knowledge. ([GitHub][15])

## Unsafe compatibility aliases

Największy znany zakaz: `ekologus.dev.proudsite.pl` nie może być canonical SEO evidence. PLAN i Goal 002 są tu dobre: final public canonical content home to `ekologus.pl`; dev URL jest preview/design/staging context. Trzeba utrzymać ten guard, bo inaczej WILQ będzie optymalizował stagingowy artefakt zamiast publicznego klienta. ([GitHub][24])

## Dead/stale plans

Ryzykowne są wszystkie kierunki sprzed aktualnej granicy: multi-client SaaS, broad RAG/vector DB, autopublish, social publishing, prompt-pack expansion. `PLAN.md` mówi, żeby ich nie dodawać przed cleanup/Goal 005 readiness; `PLANS.md` i roadmapa też trzymają te rzeczy poza aktualnym readiness claimem. ([GitHub][24])

## Gdzie kod jest zdrowszy niż docs

`apps/api/wilq_api/main.py` jest już mały. Router registry i dashboard surface registry są realnym porządkiem. `wilq/content/` jest rozdzielony na sensowne domeny: claims, drafts, handoff, knowledge, measurement, quality, revision. To jest mocniejsza baza niż typowe „AI SEO repo”. ([GitHub][8])

## Gdzie docs oversell code

Najbardziej oversellowany obszar to „Wilku-ready”. Repo często mówi o readiness cards, skills i UAT packetach, ale Goal 005 completion audit mówi: UAT incomplete, full verify not complete for the audit slice, daily content usefulness not claimable. Drugi oversell to WordPress: dry-run/draft-only readiness brzmi blisko, ale Beads mówią zero possible vendor writes i no write button. ([GitHub][3])

# 8. WHAT_NOT_TO_BUILD_NEXT

**Broad RAG/vector DB** — slop teraz. Problemem nie jest retrieval volume, tylko brak zatwierdzania i promocji wiedzy. Większy index szybciej wprowadzi review-required/private/source-mixed facts do draftów. Roadmapa wymienia broad RAG jako poza aktualnym zakresem. ([GitHub][7])

**Multi-client SaaS** — przedwczesne. `PLAN.md` mówi, że obecny depth-first workspace to Ekologus; no agency admin/billing/switcher w tym celu. Multi-client tylko pomnoży niedomknięte governance i write safety. ([GitHub][24])

**Autopublish** — absolutnie nie. Repo ma dobrą zasadę: WordPress draft-only, review-gated, zero autopublish. Autopublish teraz byłby świadomym złamaniem core safety modelu. ([GitHub][1])

**Social publishing** — nie teraz. Social inventory jest metadata-only i credentials-blocked. Publishing bez historii kanałów i duplicate guard to generator powtórek. ([GitHub][16])

**Mass article generator** — slop. Goal 005 pokazuje jednego actionable kandydata i brak approved-current knowledge. Masowe generowanie tylko zamaskuje brak wiedzy dużą liczbą draftów. ([GitHub][3])

**Generic AI writer** — zakazane przez własną architekturę. Produkcyjne generowanie ma iść przez WILQ API + SDK + Structured Outputs + strict schemas, nie przez Codex/prompty jako writer. ([GitHub][1])

**Prompt-pack skill expansion** — teraz daje fałszywy postęp. Skills są wartościowe, ale już jest 13/13 fresh/passing; brak to nie kolejny skill, tylko UAT/approved knowledge/write/measurement. ([GitHub][16])

**New dashboard screens without API-owned logic** — zły kierunek. Dashboard ma renderować view-models, nie posiadać product behavior. `ContentWorkflowSurface.tsx` już jest za duży. ([GitHub][24])

**Huge refactor without product proof** — też slop. Monolity bolą, ale refactor bez zamknięcia UAT/approved knowledge tylko przesunie problem. Następny Goal musi dotknąć dług punktowo, tylko tam gdzie blokuje product proof.

# 9. NEXT_GOAL_RECOMMENDATION

## Decyzja

Następny Goal po aktualnym Goal 005 powinien iść w **kombinację knowledge approval/promotion + Claim Ledger/generation gate**, nie w WordPress apply, Ads, measurement ani reporting.

Brutalny powód: **nie można bezpiecznie pisać, apply’ować, raportować ani optymalizować, jeśli system nadal nie ma approved-current wiedzy i nie potrafi twardo powiedzieć, które claimy wolno wygenerować.** WordPress draft apply przed tym tylko szybciej wyprodukuje ładnie opakowany slop. Ads przed tym rozproszy uwagę. Reporting przed tym będzie raportował brak operacji. Measurement przed tym będzie mierzył workflow, który nie ma zatwierdzonego contentu.

## Goal name

**Goal 006 — Approved-Current Knowledge + Claim Ledger Gate v2 for one Ekologus service**

## Outcome

Jedna wybrana usługa Ekologus — rekomenduję **BDO i sprawozdawczość środowiskowa** — przechodzi z `source_backed_review_required` do `approved_current` przez audytowany review/promotion workflow, a Content Workflow/Sales Brief/structured draft package mogą użyć wyłącznie claimów dopuszczonych przez Claim Ledger v2. BDO jest najlepszym kandydatem, bo istnieje materiał do UAT, publiczne źródło, source fact, review action i jasno opisane forbidden claims; handoff mówi jednak wprost, że nie jest to dziś aktywne zadanie do pisania i najpierw potrzebna jest decyzja Wilka. ([GitHub][28])

## Why now

Goal 005 udowodnił dokładnie tę lukę: Service Profile jest dobry do review, ale ma `approved_current=0` i production-depth zablokowane; knowledge-depth audit mówi, że initial cards były thin/internal/seeded; completion audit mówi, że no daily content usefulness. ([GitHub][11])

## Non-goals

Nie budować: WordPress apply, autopublish, Ads/Merchant writes, social publishing, broad RAG/vector DB, multi-client SaaS, mass article generator, generic writer, nowe dashboard screens bez API logic, pełny katalog usług. Goal ma dowieźć **jedną usługę** i jeden generation gate.

## Completion definition

Goal 006 jest complete tylko jeśli:

1. Jedna usługa ma approved-current source facts/cards z reviewerem, datą, freshness, source lineage, confidence i forbidden claims.
2. Review-required/private proposals nie mogą wejść do production draftu bez promotion audit.
3. Claim Ledger v2 rozróżnia allowed, blocked, review-required, measurement-required i forbidden claims.
4. Structured draft package fail-closed, jeśli brakuje approved-current service knowledge albo Claim Ledger v2.
5. Sales Brief v2 używa approved knowledge i pokazuje blokady dla niedozwolonych claimów.
6. Dashboard pokazuje source trace i claim trace bez React remap logic.
7. Wilku/owner potrafi prześledzić każde zdanie draftu do claim ledger/source connector/evidence.
8. Full verify przechodzi w realnym dev/CI env.
9. UAT/defer z Goal 005 jest zamknięte albo explicit residual risk jest zapisany przed claimem Goal 006.

## Blocked stop conditions

Stop, jeśli:

* nie ma realnego review Wilka/ownera,
* BDO/source service nie ma aktualnego publicznego lub approved private source,
* `review_required` zostaje użyte jako `approved_current`,
* claim legal/penalty/guarantee przechodzi do draftu,
* UI pokazuje rekomendację bez source connector,
* generation gate można ominąć przez Codex/skill/dashboard,
* draft package powstaje bez claim ledger,
* `ContentWorkflowSurface.tsx`, `wilq/schemas.py` albo `wilq/actions/service.py` rosną bez ekstrakcji.

# 11. GO_NO_GO

## Co można pokazać Wilkowi teraz

Można pokazać **review/blokady/traceability demo**:

* Service Profile jako ekran „co WILQ wie i czego nie wolno jeszcze użyć produkcyjnie”.
* Publiczne service review actions.
* BDO handoff jako materiał do oceny języka, źródeł, CTA i forbidden claims, nie jako aktywne zadanie pisania.
* Content queue ze statusem blocked i wyjaśnieniem, dlaczego to nie jest awaria modelu.
* Claim Ledger gate: dowód bez source connectora blokuje.
* WordPress readiness jako draft-only/dry-run, bez write button.
* Ads/GSC/Ahrefs/Localo review cards jako diagnostykę, nie optimizer apply.
* Social history inventory jako blocker, nie publishing readiness. ([GitHub][26])

## Czego nie wolno obiecać

Nie wolno obiecać:

* „WILQ jest lepszym BDOS-em”.
* „WILQ codziennie prowadzi marketing Ekologus”.
* „WILQ tworzy produkcyjny content”.
* „WILQ publikuje lub zapisuje do WordPressa”.
* „WILQ robi social publishing”.
* „WILQ optymalizuje Ads jak BDOS-grade optimizer”.
* „WILQ ma zatwierdzoną wiedzę usługową Ekologus”.
* „WILQ dowozi mierzalny uplift”.
* „Skill eval = Wilku UAT”.
* „Review-required = approved-current”.

## Czy WILQ nadaje się do codziennej pracy?

**Nie jako samodzielny daily Marketing OS.** Nadaje się do kontrolowanej sesji review i diagnozowania blokad. Nie nadaje się jeszcze jako narzędzie, które Wilku odpala codziennie i prowadzi nim content/Ads/Merchant/measurement bez developera.

## Czy WILQ nadaje się do produkcyjnego tworzenia contentu?

**Nie.** Nadaje się do review-ready draft mechanics i blocked workflow. Produkcyjny content wymaga approved-current knowledge, Claim Ledger v2, quality review, human review, audit i najlepiej UAT proof.

## Czy WILQ nadaje się do write/apply?

**Nie.** Write/apply powinien pozostać zablokowany. WordPress jest draft/dry-run/review-gated. ActionObject preview nie jest safe apply.

## Czy WILQ nadaje się do pitchowania jako „lepszy BDOS”?

**Nie.** Można pitchować jako: „BDOS-class direction / API-first Marketing OS prototype z mocnymi anti-slop gates dla Ekologus”. Claim „lepszy BDOS” wymaga minimum:

1. 3–5 real Wilku UAT sessions.
2. Co najmniej jedna usługa approved-current production-depth.
3. Claim Ledger v2 generation gate.
4. Jedna safe write/apply ścieżka draft-only z audit.
5. Measurement provenance dla przynajmniej jednego outcome.
6. Daily cockpit użyty w realnym rytmie pracy.
7. Ads albo Merchant workflow, który jest czymś więcej niż review card.

# 12. FINAL_SUMMARY_FOR_OWNER

Jesteście w dobrym kierunku, ale jeszcze nie w produkcie.

WILQ ma mocny szkielet: API-first, typed contracts, source/evidence gates, Claim Ledger, Service Profile, Content Workflow, WordPress dry-run, measurement windows, skills i dashboard surfaces. To jest dużo lepsze niż promptowy SEO writer. Ale obecny stan to **mechanically proven + review/demo-ready**, nie production-ready.

Najbardziej boli nie brak kolejnego endpointu. Najbardziej boli to, że **Ekologus knowledge nadal nie jest approved-current**, a **Wilku UAT nie jest zamknięty**. Service Profile sam mówi: `approved-current=0`, `production-depth=0`. Goal 005 completion audit sam mówi: status not complete. To jest twardy fakt, nie opinia. ([GitHub][3])

Następny Goal nie powinien być WordPress apply, Ads optimizer ani social. Następny Goal powinien dowieźć **jedną zatwierdzoną usługę Ekologus + Claim Ledger v2 generation gate**. Dopiero wtedy ma sens draft apply, measurement i Ads depth.

Absolutnie nie claimujecie: production-ready, daily Marketing OS, lepszy BDOS, safe write/apply, autopublish, social publish, measured uplift, zatwierdzona wiedza Ekologus.
